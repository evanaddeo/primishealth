# AI Agent Instructions — Primis

This file is for AI coding agents working in this repository. Read it before writing any code.

---

## Agent Boundary Rules

### Rule 1 — Implement only the currently assigned commit unit

You are authorized to implement exactly one commit unit (CU) per session. The CU is defined in the
handoff prompt or in `plans/phase-a-repo-tooling-foundation.md` (Phase A) and in
`docs/source-of-truth/primis_full_implementation_spec_commit_plan.md`.

Do not implement adjacent CUs, future phase work, or "nice to have" improvements that are not
explicitly in scope. Do not start the next CU until the current CU's acceptance criteria and
verification commands all pass.

### Rule 2 — Do not invent schema, formulas, or styles not in source docs

You must not invent:

- Database table names, column names, or metric codes not in `primis_data_model_health_metric_schema.md`
- Score formulas, baseline thresholds, or algorithm logic not in `primis_scoring_algorithms_spec.md`
- UI colors, typography, spacing, or component patterns not in `primis_ui_ux_design_system_spec.md`
- AI prompt structures, intent types, or context packet shapes not in `primis_ai_context_engine_spec.md`
- AWS service choices or architecture patterns not in `primis_technical_architecture_document.md`

If a source doc is silent on a detail you need, leave a `// TODO(ADR):` comment and create a decision
record under `docs/decisions/` rather than guessing.

### Rule 3 — Read the source docs before implementing

Before writing any code for a commit unit, read the sections listed in the handoff prompt and in the
Phase A plan (`plans/phase-a-repo-tooling-foundation.md §2`). The required reading order is:

See `docs/README.md` for one-sentence descriptions of each document's authority domain.

### Rule 4 — Add tests for every non-trivial change

- Every non-trivial function, data transformation, or utility must have a corresponding test.
- Tests live in `*.test.ts` files co-located with the source or in a `test/` subdirectory.
- Use Vitest for all TypeScript tests.
- Tests must be deterministic — no real network calls, no real database connections in unit tests.
- All fixtures must be redacted (see `database/fixtures/README.md` when it exists).
- See `tests/README.md` for test conventions (created in CU-005).

### Rule 5 — Do not commit secrets or real provider payloads

Never commit:

- `.env` or `.env.local` files
- Real API keys (OpenAI, Anthropic, AWS, Google OAuth)
- Real OAuth tokens or refresh tokens
- Real user IDs, email addresses, or device identifiers
- Unredacted provider API payloads

`.env.example` contains placeholder values only and is safe to commit. Real credentials remain
`PLACEHOLDER` throughout Phase A. Provider payload fixtures must be redacted before committing
(see `database/fixtures/README.md` when it exists).

## Workflow

1. Read this file and the required source docs for your assigned CU.
2. Read the CU definition in `plans/phase-a-repo-tooling-foundation.md` (for Phase A CU-001–007).
3. Implement only the files listed under "Files created / edited" for that CU.
4. Run the verification commands listed in the CU plan.
5. Confirm all acceptance criteria pass.
6. Use the exact commit message format: `<area>: <short imperative summary> (<CU-ID>)`.
7. Use the exact branch name format: `cu/<cu-id-lowercase>-<short-name>`.

See `CONTRIBUTING.md` for full workflow details, area prefixes, and ADR instructions.

## Out-of-Scope Work for Phase A (CU-001–007)

Do not implement any of the following during Phase A:

- Mobile app scaffolding (`expo init`, `create-expo-app`, or anything in `apps/mobile/`)
- Database schema, migration files, or seed data
- Provider connectors (Google Health, Fitbit, Apple HealthKit)
- Scoring algorithms, baselines, or data-quality calculations
- AI gateway, prompts, or context packets
- UI components, design tokens, or screens
- AWS infrastructure (CDK stacks, RDS, Lambda, etc.)
- Any Phase B (CU-008+) or later work

If you think something is needed but it is not in the current CU scope, leave a comment and defer it.
Do not implement it speculatively.
